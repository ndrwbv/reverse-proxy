# Фича: Автоматический деплой reverse-proxy

**Ветка:** `main`
**Создана:** 2026-03-25
**Статус:** реализовано

## Контекст

**Зачем:** Сейчас reverse-proxy деплоится вручную — нужно зайти по SSH на сервер, сделать `git pull`, проверить конфиг и перезагрузить nginx. Оба проекта (bus-schedule, reservation-service) уже деплоятся автоматически через GitHub Actions, а reverse-proxy — единственная репа без CI/CD. Это неконсистентно и создаёт риск забыть задеплоить изменения конфигов.

**Что есть сейчас:** Ручной деплой через SSH:
```bash
ssh root@<IP> "cd /opt/reverse-proxy && git pull && docker-compose exec nginx nginx -s reload"
```

## User Stories

### US-1 — Автодеплой при пуше в main (Приоритет: P1)

Как разработчик, я хочу чтобы при пуше в `main` изменения nginx-конфигов и docker-compose автоматически доезжали на сервер и применялись, чтобы не ходить по SSH вручную.

**Почему этот приоритет:** Основная цель фичи — убрать ручной деплой.

**Как проверить независимо:** Пушнуть изменение в nginx-конфиг, убедиться что workflow отработал и конфиг обновился на сервере.

**Сценарии приёмки:**

1. **Дано** изменён `nginx/conf.d/severbus.conf`, **Когда** пуш в `main`, **Тогда** GitHub Actions синхронизирует файлы на сервер и перезагружает nginx
2. **Дано** изменён `docker-compose.yml`, **Когда** пуш в `main`, **Тогда** GitHub Actions синхронизирует файлы и делает `docker-compose up -d`

---

### US-2 — Ручной запуск деплоя (Приоритет: P1)

Как разработчик, я хочу иметь возможность запустить деплой вручную через GitHub Actions (workflow_dispatch), чтобы в экстренных случаях можно было передеплоить без пуша.

**Сценарии приёмки:**

1. **Дано** текущий код в `main`, **Когда** запускаю workflow вручную в GitHub Actions, **Тогда** происходит деплой

---

### US-3 — Проверка конфига перед применением (Приоритет: P1)

Как разработчик, я хочу чтобы перед reload nginx проверялся синтаксис конфига (`nginx -t`), чтобы битый конфиг не уронил прокси.

**Сценарии приёмки:**

1. **Дано** корректный nginx-конфиг, **Когда** деплой, **Тогда** `nginx -t` проходит, nginx перезагружается
2. **Дано** некорректный nginx-конфиг, **Когда** деплой, **Тогда** `nginx -t` падает, workflow фейлится, nginx НЕ перезагружается

---

## Ключевые сущности

- **GitHub Actions workflow** (`deploy.yml`): rsync файлов на сервер + перезапуск/reload nginx
- **Серверный путь**: `/opt/reverse-proxy` — куда синхронизируются файлы
- **Секреты GitHub**: `SSH_PRIVATE_KEY`, `DEPLOY_HOST`, `DEPLOY_USER` — те же что в других репах

## Edge Cases

- Что если на сервере есть `certbot/` с сертификатами, а rsync с `--delete` его удалит? → Нужен `--exclude='certbot/'`
- Что если nginx-конфиг синтаксически валиден, но ломает маршрутизацию? → Health check после reload
- Что если docker-compose.yml изменился и нужен `up -d`, а не просто reload? → Всегда делать `up -d` + reload

## Критерии приёмки

- [ ] В репе `reverse-proxy` есть `.github/workflows/deploy.yml`
- [ ] Workflow запускается на push в `main` и по `workflow_dispatch`
- [ ] Файлы синхронизируются через rsync с исключением `certbot/`, `.git`
- [ ] Перед reload выполняется `nginx -t` — если конфиг битый, workflow падает
- [ ] После rsync выполняется `docker-compose up -d` (пересоздаёт контейнеры если compose изменился)
- [ ] После `up -d` выполняется `nginx -s reload` (применяет изменения конфигов без даунтайма)
- [ ] Health check: `curl` на оба домена (`severbus.ru`, `slotik.tech`)
- [ ] Документация обновлена: `context.md`, `README.md`
